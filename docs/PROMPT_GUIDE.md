# 프롬프트 관리 가이드

Senior Reviewer의 프롬프트 파일을 추가/수정하는 방법을 안내합니다.
이 문서를 AI 코딩 어시스턴트에게 제공하면 프롬프트 작업을 도와줍니다.

---

## 프롬프트 구조

```
prompts/
├── agents/          ← Layer 1: 에이전트 기본 역할 (필수, 범용)
│   ├── bug-detector.md
│   ├── security-checker.md
│   ├── code-quality.md
│   └── synthesizer.md
├── teams/           ← Layer 2: 팀별 공통 규칙 (선택)
│   └── frontend.md
└── stacks/          ← Layer 3: 기술 스택별 규칙 (선택, 복수 가능)
    ├── nextjs-app-router.md
    ├── react-performance.md
    ├── tanstack-query.md
    ├── tailwindcss.md
    └── turborepo.md
```

각 에이전트의 system prompt는 `agents/*.md` + `teams/{team}.md` + `stacks/{stack}.md`를 `---` 구분자로 연결하여 합성됩니다.

## 새 스택 프롬프트 추가하기

대상 레포에서 사용하는 기술 스택에 대한 프롬프트 파일이 없을 때, 새로 만드는 절차입니다.

### Step 1. 해당 기술에 적합한 Best Practice 소스 찾기

Claude Code의 `find-skills` 기능을 사용하여 해당 기술에 대한 BP(Best Practice) skill이 있는지 확인합니다.

```
/find-skills {기술명} best practices
```

적합한 skill이 있으면 설치하고, skill의 내용을 검토하여 프롬프트에 반영할 핵심 규칙을 추출합니다.

### Step 2. 프롬프트 파일 작성

`prompts/stacks/{기술명}.md` 파일을 생성합니다.

#### 작성 원칙

1. **코드 리뷰에 필요한 규칙만 포함**
   - 초기 설정, 설치 방법 등은 제외
   - "이 코드는 문제가 있다"고 판단할 수 있는 규칙만 포함

2. **안티패턴 중심으로 작성**
   - "이렇게 하세요" (튜토리얼)가 아니라 "이렇게 하면 안 됩니다" (리뷰 기준)
   - 각 규칙에 왜 안 되는지(이유)와 어떻게 해야 하는지(대안)를 함께 기술

3. **구체적인 코드 패턴 포함**
   - Bad/Good 예시가 있으면 리뷰 정확도가 올라감
   - 단, 너무 길어지지 않도록 핵심만

4. **범용성 유지**
   - 특정 프로젝트에만 해당하는 규칙은 넣지 않음
   - 해당 기술을 사용하는 어떤 프로젝트에서든 적용 가능해야 함

5. **분량 조절**
   - 너무 짧으면 리뷰 품질 저하, 너무 길면 토큰 낭비
   - 100~300줄 정도가 적당 (현재 stacks 파일들 기준)

#### 파일 구조 템플릿

```markdown
# {기술명} 리뷰 규칙

## 1. {카테고리명}

### {규칙 제목}

{왜 이것이 중요한지 설명}

Bad:
```코드```

Good:
```코드```

...
```

### Step 3. 문서 업데이트

프롬프트 파일을 추가하면 아래 문서들도 함께 업데이트해야 합니다:

- [ ] `README.md` — "사용 가능한 프롬프트" 테이블에 새 항목 추가
- [ ] `docs/SETUP_GUIDE.md` — stacks 테이블에 새 항목 추가
- [ ] `docs/architecture.html` — Stack 프롬프트 영역의 file-chip에 새 항목 추가
- [ ] `caller-templates/review-caller.yml` — 필요한 경우 예시 업데이트

### Step 4. 빌드 & 커밋

```bash
pnpm build   # 빌드 확인 (프롬프트 파일은 런타임에 읽히지만, 코드 변경이 있을 수 있으므로)
```

커밋 후 main에 머지하면 해당 스택을 지정한 모든 대상 레포에 즉시 반영됩니다.

## 새 팀 프롬프트 추가하기

새 팀(예: backend)을 추가하는 절차입니다.

### Step 1. 프롬프트 파일 작성

`prompts/teams/{팀명}.md` 파일을 생성합니다.

#### 작성 원칙

1. **팀 공통 코딩 컨벤션 중심**
   - 해당 팀에서 공통으로 따르는 코딩 규칙
   - 아키텍처 패턴, 네이밍 규칙, 코드 구조 등

2. **기술 스택 독립적**
   - 특정 라이브러리/프레임워크에 한정된 규칙은 `stacks/`에
   - 팀 프롬프트에는 기술에 무관한 팀 규칙만

3. **agents/*.md와 중복 방지**
   - 에이전트 프롬프트에 이미 있는 범용 규칙은 반복하지 않음
   - 팀 특화 규칙만 추가

### Step 2. 문서 업데이트

스택 추가 시와 동일:
- [ ] `README.md`, `SETUP_GUIDE.md`, `architecture.html` 업데이트

---

## 에이전트 프롬프트 수정하기

`prompts/agents/*.md` 파일을 수정할 때 주의사항:

1. **범용성 유지**: FE/BE 특화 내용을 넣지 않음. 모든 팀이 공유하는 프롬프트
2. **리뷰 톤 가이드라인 유지**: 각 에이전트에 "리뷰 작성 가이드" 섹션이 있음. 수정 시 톤 일관성 유지
3. **synthesizer 동기화**: 에이전트 출력 형식을 바꾸면 synthesizer 프롬프트도 함께 수정

---

## 스택 파일이 필요한데 없을 때

대상 레포에 중요한 기술 스택인데 `prompts/stacks/`에 해당 파일이 없는 경우:

1. 사용자에게 해당 스택의 프롬프트 파일이 없음을 알립니다
2. `find-skills`로 해당 기술의 BP skill을 검색합니다
3. 적합한 skill이 있으면 내용을 검토하고 핵심 규칙을 추출합니다
4. 위 "새 스택 프롬프트 추가하기" 절차에 따라 파일을 생성합니다
5. 관련 문서를 모두 업데이트합니다

> 프롬프트 품질이 리뷰 품질에 직접 영향을 줍니다. 무조건 짧게 줄이기보다는, 리뷰에 필요한 규칙을 꼼꼼히 검토하여 포함하세요.
